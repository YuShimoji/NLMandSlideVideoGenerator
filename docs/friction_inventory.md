# 摩擦インベントリ: 1本完走の体験逆算

最終更新: 2026-03-22
対象体験: YouTube公開の長尺解説動画 (20-30分) を週1本以上制作

---

## パイプライン全体像

根本ワークフロー (DESIGN_FOUNDATIONS.md Section 0) に準拠:

```
Phase 0: NotebookLM (人間操作)
  [ソース投入] → [Audio Overview生成] → [音声再投入+テキスト化]

Phase A: Python自動パイプライン (~2-5分)
  [NLMテキスト入力] → [Gemini構造化(speaker/text)] → [画像調達] → [スライド生成] → [CSV合成]

Phase B: YMM4手動操作 (~30-120分)
  [CSVインポート] → [Voice生成] → [プレビュー確認] → [BGM配置] → [レンダリング]

Phase C: 後処理 (~5分)
  [MP4検証] → [メタデータ生成] → [YouTube公開(未実装)]
```

## 摩擦ランキング (影響度順)

### CRITICAL: 週1本ペースを阻害する摩擦

#### F-001: Geminiクォータ枯渇 (20req/day) — **解消済み**
- **影響**: 2本目以降のバッチ実行でMockProvider落下 → 台本品質崩壊
- **解消**: SP-043 Phase 1-4 完了。ILLMProvider 5プロバイダー (Gemini/OpenAI/Anthropic/Groq/モック) + Streamlit設定UI
- **状態**: `LLM_PROVIDER` 環境変数でプロバイダー切替可能

#### F-002: YMM4手動操作が全体の80%を占める
- **影響**: Phase Bの30-120分が完全手動。週3本ペースでは週6-24時間
- **現状対策**: なし (YMM4にCLIモードが存在しない)
- **残課題**: YMM4 SDK/APIの制限。根本解決は困難
- **解消コスト**: 高 (YMM4側の制約)

#### F-003: レンダリング時間が動画長の2-5倍
- **影響**: 30分動画 → 60-150分のレンダリング待ち
- **現状対策**: 就寝時に実行する運用回避
- **残課題**: マシンスペック依存。バッチレンダリング未対応
- **解消コスト**: 低 (運用で回避可能)

### HIGH: 品質に影響する摩擦

#### F-004: フォールバック時の品質低下が非可視 — **解消済み**
- **影響**: MockProvider/TextSlideへの落下をユーザーが気づかない → 低品質動画が完成
- **解消**: PipelineStats fallback_events + FALLBACK WARNING 表示実装済み

#### F-005: セグメント粒度が制御不能 — **解消済み**
- **影響**: 30分指定でも実尺は台本品質次第。過少→スカスカ、過多→早口
- **解消**: SP-044 Phase 1-3 完了。推定尺計算 + セグメント検証 + LLM自動調整 + CLI手動モード

#### F-006: クレジット未挿入でYouTube公開 — **解消済み**
- **影響**: Pexels/Pixabayライセンス違反のリスク
- **解消**: パイプライン→metadata.json + image_credits 自動統合実装済み

### MEDIUM: 効率に影響する摩擦

#### F-007: TextSlide生成が遅い
- **影響**: 100セグメントで3-5分 (並列化未実装)
- **現状対策**: なし
- **残課題**: PIL処理の並列化 or キャッシュ強化
- **解消コスト**: 中

#### F-008: API key設定の複雑さ
- **影響**: 新環境セットアップ時に6種のAPIキー設定が必要
- **現状対策**: .env.example + ENVIRONMENT_SETUP_GUIDE.md
- **残課題**: verify_api_keys() は存在するが起動時自動実行ではない
- **解消コスト**: 低

#### F-009: 手動プレビュー確認の負荷
- **影響**: 30分動画の全セグメントを目視確認 → 30-60分
- **現状対策**: なし
- **残課題**: 自動品質スコアリング or サンプル抽出確認
- **解消コスト**: 高

## 体験フロー別の改善余地

### Phase A → Phase B の受け渡し
- CSV生成後、ユーザーがYMM4を手動起動しCSVパスを手入力
- **改善案**: CSV生成完了時にYMM4起動コマンド+CSVパスをクリップボードにコピー

### Phase B内の効率化
- BGM配置はstyle_template.jsonから自動適用可能 (SP-031実装済み)
- Voice生成は自動ON可能 (SP-024実装済み)
- **残る手動**: プレビュー確認 + レンダリング開始 (最低2操作)

### Phase C → YouTube公開
- メタデータ生成は自動 (SP-038 Phase 1-2)
- クレジット統合は配線のみ
- YouTube APIアップロードはOAuth未整備 (SP-038 Phase 3)

## 数値で見る摩擦

週1本 (30分動画) の場合:
- Phase 0 (NLM手動): 10-30分
- Phase A (自動): 2-5分 → 問題なし
- Phase B (手動): 30-120分 → **最大の時間消費**
- Phase C (後処理): 5分+レンダリング60-150分
- **合計**: 107-310分/本 (1.8-5.2時間)

週3本を目指す場合:
- Phase 0 x3: 30-90分 → **NLM操作が新たなボトルネック**
- Phase A (バッチ): 10-15分 → 問題なし
- Phase B x3: 90-360分 → **4.5-6時間/週** (Phase Bだけで)
- **改善ポイント**: Phase B手動時間の半減 + Phase 0のバッチ化で週3本が現実的
