# 動画制作者GUI仕様

SP-053 | Status: draft | Updated: 2026-03-23

---

## 目的

動画制作者が「トピック選定からYouTube公開」までの全フローを、
単一のGUIから快適に操作できる統合ダッシュボードを提供する。

### 設計原則

1. **ワークフロー中心**: 機能別ページではなく、制作フローの段階に沿った画面遷移
2. **最小操作**: 各段階で制作者が行う操作を最小化。自動化できる箇所は自動化
3. **進捗可視化**: 全制作ラインの状態を一覧で把握可能
4. **バッチ選定+個別制作**: SP-050確定の2段階制作フローに準拠

### 前提 (DESIGN_FOUNDATIONS準拠)

- 入力層: NotebookLM (人間操作 or Playwright半自動化)
- 変換層: Python + Gemini (自動)
- 出力層: YMM4 (人間操作、ほぼノータッチ目標)

### 画面 1: 新規ライン作成 / バッチ選定

**目的**: 複数トピックをまとめて評価し、Go/No-Go を判定する

```
┌─────────────────────────────────────────────────────────────┐
│  [< ボード] バッチ選定                                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─ トピック入力 ──────────────────────────────────────┐   │
│  │ [トピック名]  [ソースURL/テキスト貼り付け]           │   │
│  │ [+ トピック追加]  [RSSから取得]                       │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                             │
│  ┌─ NLM操作支援 ──────────────────────────────────────┐   │
│  │ [NLMを開く (Playwright)]  [音声ファイルを指定]       │   │
│  │ Status: Audio Overview生成完了 (12:34, 8分32秒)      │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                             │
│  ┌─ LEGACY: AI評価 (廃止) ────────────────────────────┐   │
│  │ ※ Gemini動画適性スコアリングは廃止 (2026-03-23)      │   │
│  │ ※ トピック選定は人間判断。Go/No-Goは手動で行う        │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                             │
│  [選定を確定 → 構造化フェーズへ]                             │
└─────────────────────────────────────────────────────────────┘
```

#### 操作フロー

1. トピックを1-5件入力 (手動 or RSS取得)
2. 各トピックについてNotebookLMでAudio Overview生成 (手動 or Playwright支援)
3. 音声ファイルを指定
4. ~~AI評価を実行 (Gemini: トピックの動画適性スコア+コメント)~~ **LEGACY: 廃止。人間がトピック選定を判断**
5. Go/No-Go判定 (人間判断)
6. 確定 → Goしたラインが「構造化」列に移動

## データモデル: ProductionLine

各制作ラインの状態を管理するデータ構造。

```python
@dataclass
class ProductionLine:
    line_id: str                    # UUID
    topic: str                      # トピック名
    created_at: datetime            # 作成日時
    status: LineStatus              # enum: draft/selecting/structuring/producing/reviewing/publishing/done
    current_phase: int              # 0-7

    # Phase 0-1: NLM
    source_urls: list[str]          # ソースURL
    audio_path: str | None          # Audio Overview音声ファイル
    transcript_path: str | None     # テキスト化ファイル

    # Phase 2-3: 構造化
    script_json_path: str | None    # 構造化台本
    segment_count: int              # セグメント数
    estimated_duration: float       # 推定尺(秒)

    # Phase 4-5: スライド+CSV
    slides_dir: str | None          # スライド画像ディレクトリ
    images_dir: str | None          # 素材画像ディレクトリ
    csv_path: str | None            # 最終CSV
    metadata_path: str | None       # metadata.json

    # Phase 6: YMM4
    ymm4_project_path: str | None   # YMM4プロジェクトファイル
    mp4_path: str | None            # レンダリング済みMP4

    # Phase 7: 公開
    thumbnail_path: str | None      # サムネイル画像
    youtube_url: str | None         # YouTube URL

    # AI評価 — LEGACY: 廃止 (2026-03-23)。トピック選定は人間判断。
    ai_score: float | None          # LEGACY: 常にNone
    ai_comment: str | None          # LEGACY: 常にNone
    go_decision: bool | None        # 人間が手動で判定

    # メタ
    topic_dir: str                  # data/topics/{topic_id}/
    error_log: list[str]            # エラー履歴
```

保存先: `data/production_lines.json`

## 技術選定

### Phase 1: Streamlit再構成 (即時)

既存のStreamlit UIを制作者フローに合わせて再構成する。

**理由**:
- 既存の8ページ分のUIコード資産を活用
- バックエンドとの接続が既に実装済み
- 新規フレームワーク学習コスト不要

**制約**:
- リアルタイム更新が弱い (ポーリングベース)
- 状態管理がsession_stateに依存 (永続化は自前実装)

### Phase 2: Playwright統合 (中期)

NotebookLM Web UIのPlaywright半自動化。

**方針**:
- persistent browser contextでGoogle認証を再利用
- codegen記録 → 半自動スクリプト化
- 制作者が操作を見守りながら介入可能な「監視モード」

### Phase 3: 専用GUI検討 (長期, 必要に応じて)

Streamlitの制約が制作体験を阻害する場合、以下を検討:
- Electron + React (desktop/に骨格あり)
- Tauri + Svelte
- FastAPI + Next.js

## 既存UIとの関係

### 統合方針

既存の機能別ページは「詳細操作」として残し、制作者GUIから遷移可能にする。

| 既存ページ | 新GUIでの位置づけ |
|---|---|
| ホーム | プロダクションボードに置換 |
| リサーチ・台本照合 | ライン詳細 → 台本レビューに統合 |
| 素材パイプライン | ライン詳細 → Phase 2-5の自動実行に統合 |
| AI生成 (Pipeline) | 廃止候補 (旧パイプライン前提) |
| バッチ制作キュー | バッチ選定画面に進化 |
| 生成物一覧 | ライン詳細 → 成果物プレビューに統合 |
| ドキュメント | そのまま残す (開発者向け) |
| 設定 | そのまま残す |
| テスト | そのまま残す (開発者向け) |

## NLM自動化

### 技術選定 (調査結果 2026-03-23)

**推奨: PyNotebookLM (pynotebooklm) ライブラリ直接統合**

| 方式 | 利点 | リスク | 優先度 |
|---|---|---|---|
| PyNotebookLM (v0.21.0) | 31機能搭載、Audio Overview+スライド生成対応、即利用可能 | 内部API依存、2-4週間のクッキー更新 | Phase 3a (推奨) |
| Custom Playwrightスクリプト | 細かい制御可能 | UIセレクタ脆弱、メンテコスト高 | Phase 3b (補完) |
| NLM内部API直呼出 | 最も安定 | リバースエンジニアリング必要 | Phase 3c (将来) |

### PyNotebookLM統合計画

```bash
pip install pynotebooklm
playwright install chromium
pynotebooklm auth login      # 初回: ブラウザログイン
pynotebooklm auth check      # 認証確認
```

統合対象機能:
- ノートブック作成 + ソース追加 (URL/YouTube/テキスト)
- Audio Overview生成 (ポッドキャスト形式)
- スライド生成 (NLMスライド機能、PIL代替)
- テキスト化 (音声→テキスト)

### 対象操作

| 操作 | 自動化可能性 | 方式 |
|---|---|---|
| NotebookLMにログイン | 半自動 | pynotebooklm auth login (persistent context) |
| ソース投入 (URL) | 自動 | pynotebooklm API |
| ソース投入 (テキスト/PDF) | 自動 | pynotebooklm API |
| Audio Overview生成 | 自動 | pynotebooklm API |
| 音声ダウンロード | 自動 | pynotebooklm API |
| スライド生成 | 自動 | pynotebooklm API (Phase 3スライドの代替候補) |
| テキスト化 (音声再投入) | 半自動 | pynotebooklm + Gemini Audio API (SP-051) |

### 監視モード

制作者がPlaywrightブラウザを見ながら、自動操作を監視・介入できるモード。

```
[NLM操作支援パネル]
Status: Audio Overview生成中... (残り約2分)
[一時停止]  [手動で引き継ぐ]  [スキップ]
```

### 技術的課題

- PyNotebookLMは内部API依存 → NLM側の変更で停止リスク
- Google認証クッキーは2-4週間で更新が必要
- Google ToS: 自動化がGoogleの利用規約に抵触しないか要確認
- Audio Overview生成の所要時間は未実測 (推定3-10分)
